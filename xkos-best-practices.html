The Best Practices

Description of a statistical classification

§5.1 of the XKOS specification states that

“In order to represent classifications and classification schemes, XKOS does not define specific classes but uses directly SKOS classes, adding to them different properties that specifically apply to classifications or classification schemes. Each major version of a classification is represented in XKOS by a `skos:ConceptScheme`.”
.

This best practices document gives further recommendations on how best to describe a `skos:ConceptScheme` that identifies a statistical classification, with mandatory, recommended and optional properties. Note that some constraints impose the presence of a property in English, hence the "lang" column with value "en" for certain properties

Mandatory, Recommended and Optional properties to describe statistical classifications
Property Lang. Card. Range Description
Mandatory properties
`skos:notation` 1..1 `xsd:string` The value must be the short name of the classification, with no language tag.
`skos:prefLabel` en 1..1 `rdf:langLiteral` The value must be the complete name of the classification scheme in english.
`dcterms:issued` 1..1 `xsd:date` The value must be the publication date of the classification scheme
Recommended properties
`skos:prefLabel` any language 0..n `rdf:langLiteral` The value must be the complete name of the classification scheme in another language than english, with a proper language tag.
`dc:description` en 0..1 `rdf:langLiteral` The value must be a short descriptive text about the classification scheme in english.
`skos:scopeNote` en 0..1 `xkos:ExplanatoryNote` The explanatory note MUST have a `xkos:plainText` property which value is a long descriptive text about the classification scheme in English, with a language tag set at ‘@en’
`dcterms:modified` 0..1 `xsd:date` The value must be the last modification date of the classification scheme.
`xkos:belongsTo` 0..1 `skos:Concept` Connects a major version of a classification to a concept representing the overall classification.
`xkos:levels` 0..1 `rdf:List` Lists the classification levels represented as an RDF list of ordered levels (instances of `xkos:ClassificationLevel`).
`xkos:covers` 0..1 `skos:Concept` The domain covered by the classification or its levels (eg. activities, products, etc.). Subproperties `xkos:coversExhaustively` and `xkos:coversMutuallyExclusively` can also be used.
`xkos:coversExhaustively` 0..1 `skos:Concept` The domain covered by the classification (exhaustive coverage).
`xkos:coversMutuallyExclusively` 0..1 `skos:Concept` The domain covered by the classification (mutually exclusive coverage).
`xkos:variant` 0..n `skos:ConceptScheme` `Connects a classification scheme to its variant(s)`
`xkos:numberOfLevels` 0..1 `xsd:integer` Depth of a classification (number of hierarchically imbricated levels) ; 1 for a flat classification.
`xkos:follows` 0..1 `skos:ConceptScheme` Links major revisions of classifications.
`xkos:supersedes` 0..1 `skos:ConceptScheme` Links major revisions of classifications.
Optional properties
`skos:scopeNote` any language 0..n `xkos:ExplanatoryNote` Each explanatory note MUST have a xkos:plainText property which value is a long descriptive text about the classification scheme in another langage than english, with the langage tag set appropriately
`dcterms:language` 0..n `xsd:language` Indicates the langage coverage of the classificaiton scheme. Each value must correspond to a possible langage of labels within that classification.
`insee:legalMaterial` 0..n `rdf:Resource` Legislation related to the nomenclaure. Use of an ELI URI is preferred, if one is available ([[ELI]])

Publishing statistical classification data

While the XKOS specification introduces useful properties to describe statistical classifications, it does not give recommendations on how to publish statistical classification data. This section explains how to use [[DCAT]] to describe and publish the datasets of statistical classifications and correspondences.

Datasets of classifications

It is recommended to use DCAT V2 to describe and disseminate metadata associated with classifications.

Each revision of a classification should correspond to one `dcat:Dataset`. For instance, in the example below, the classification NAF Rev1 and NAF Rev2 are separated in distinct `dcat:Dataset`s with different version information. Each classification’s dataset belongs to a `dcat:Catalog`.

Each dataset should have downloadable files in the necessary formats for both human and machine agents. In particular, statistical classifications should be provided for download in an RDF file based on [[SKOS]] and [[XKOS]] vocabularies, containing the full content of the classification. In the example, CSV and RDF (SKOS) formats are provided. The `dct:conformsTo` property should be set to the URI of the [[XKOS]] vocabulary : http://rdf-vocabulary.ddialliance.org/xkos.

For practical reasons it is important to have direct access to the latest version of a classification for directly retrieving the updated notes. However, for reasons of nomenclature expertise it is also important to have access to changes in notes that have affected the nomenclature. It is therefore recommended to have access to the history of modifications of all notes within the revision of the classification included in a `dcat:Dataset`. These versions containing the history of all notes are considered as different distributions of the datasets. They should be described by specific properties. Here we use `rdfs:comment` for describing and discriminating the distributions, given a lack of other alternatives. Other information is given by the `dct:PeriodOfTime` that is wider in the case of the distribution with all the note changes.

Datasets of correspondences

Each correspondence table shall also be provided as a distinct Dataset in the catalog. This dataset can be linked with the dataset of the source classification and the dataset of the target classification with :

  • A generic `dct:relation` property, that is expressed both from the source and target dataset to the correspondence table dataset, and from the correspondence dataset to its source and target.
  • A `dcat:Relationship` that indicates more precisely the role of each dataset, either source or target in the correspondence table. The role of each dataset is captured by the `dcat:hasRole` property, with one of these 2 values :
    • `xkos:correspondenceFromRole`
    • `xkos:correspondenceToRole`
    The exact URIS for roles have been proposed to XKOS but are not part of the standard at the time of writing of this document, and so are subject to change.

The example below presents the correspondence between Naf rev2 and Naf rev1 classification.

# The correspondence as a dcat:Dataset, with links to the source and target datasets
# from the previous example
ex:CorrespondenceTableNafrev1Nafrev2Dataset a dcat:Dataset ;
  dcat:distribution ex:CorrespondenceTableCsv ;
  dcat:distribution ex:CorrespondenceTableRdf ;
  # Qualified relation to the source dataset
  dcat:qualifiedRelation [
    a dcat:Relationship ;
    dct:relation ex:NafRev1Dataset ;
    dcat:hadRole xkos:correspondenceFromRole ;
  ] ;
  # Qualified relation to the target dataset
  dcat:qualifiedRelation [
    a dcat:Relationship ;
    dct:relation ex:NafRev2Dataset ;
    dcat:hadRole xkos:correspondenceToRole ;
  ] ;
.

URI patterns Best Practices

Proposed URI pattern for classifications

{scheme}://{authority}/{all_classifications}/{classification_naming_context}/{concept_scheme_id}

Where:

  • `{scheme}`: is the URI scheme, typically “http” or “https”.
  • `{authority}` : the domain name of the authority attributing the URI to the resource. This guarantees the unicity of the URI at Web scale. Note that the domain name can be a dedicated subdomain of the more general domain name of the authority e.g. “data.acme.org” or “id.acme.org”.
  • `{all_classifications}` : is a fixed component in the URI indicating that what follows pertains to the identification of an entity in the scope of a classification, as opposed to an entity in another domain, like statistical data, document metadata, etc. Typical examples are “codes”, “classifications”, etc.
  • `{classification_naming_context}` : an identifier for the naming context that will contain the different components of this classification; This component typically is based on the classification id in one of its specific revisions. Examples are “nafr2”, “pcs1982”, etc.
  • `{concept_scheme_id}` : an identifier for the concept scheme that represents a revision of the classification.

Given that the french NAF rev2 is identified with the classification_naming_context `nafr2` and the concept_scheme_id `naf`, the corresponding URI would be `http://id.insee.fr/codes/nafr2/naf`

Proposed URI pattern for classification levels

{scheme}://{authority}/{all_classifications}/{classification_naming_context}/{level}

Where:

  • `{scheme}`, `{authority}`, `{all_classifications}` and `{classification_naming_context}` components have the same meaning as the in the URI pattern for the classification;
  • `{level}` : an identifier for a level in this classification. Some implementations have chosen the plural forms, e.g. “sections”, “classes”, “divisions”, etc.

Given that the french NAF rev2 contains levels that are “sections”, “divisions”, “groupes”, “classes”, “sousClasses”, the corresponding URI would be :

  • http://id.insee.fr/codes/nafr2/sections
  • http://id.insee.fr/codes/nafr2/divisions
  • http://id.insee.fr/codes/nafr2/groupes
  • http://id.insee.fr/codes/nafr2/classes
  • http://id.insee.fr/codes/nafr2/sousClasses

Proposed URI pattern for classification entries (Concepts)

{scheme}://{authority}/{all_classifications}/{classification_naming_context}/{level}/{concept_id}

Where:

  • `{scheme}`, `{authority}`, `{all_classifications}` and `{classification_naming_context}` components have the same meaning as in the URI pattern for the classification;
  • `{level}` : an identifier for the level of the concept. It is key to understand that the level is an intrinsic part of the concept identity in statistical classifications. A concept cannot be moved in a given revision of a classification. Hence the level is part of the concept URI. Implementations have chosen to use the singular form of the level, thus a different stem from the URI of the corresponding level. Examples are “section”, “class”, “division”, etc.
  • `{concept_id}` : an identifier for the specific concept within the classification; the format of the identifier varies depending on the classification and the level. Examples are “A”, “1963”, “10.73”, etc.

Given that the french NAF rev2 contains levels identified with the level codes “section”, “division”, “groupe”, “classe”, “sousClasses”, corresponding examples of concepts at each of these levels are :

  • http://id.insee.fr/codes/nafr2/section/A
  • http://id.insee.fr/codes/nafr2/division/01
  • http://id.insee.fr/codes/nafr2/groupe/01.1
  • http://id.insee.fr/codes/nafr2/classe/01.16
  • http://id.insee.fr/codes/nafr2/sousClasse/01.16Z

Proposed URI pattern for Labels

{concept_URI}/{label_id_length_language}

Where:

  • `{concept_URI}` : the concept URI, as described above. The URI for the label is hierarchically below the URI of the concept having this label, since a label entity is attached to a single concept.
  • `{label_id_length_language}` : a component identifying the label, typically composed with a fixed part, the label length and the languages. Examples are “label40fr” or “label65en”

  • http://id.insee.fr/codes/nafr2/classe/01.16/label40fr
  • http://id.insee.fr/codes/nafr2/classe/01.16/label65fr

Proposed URI pattern for Notes (versioned)

{concept_URI}/{type_of_note}/{note_version}/{language}

Where:

  • `{concept_URI}` : the concept URI, as described above. The URI for the note is hierarchically below the URI of the concept having this note, since a note entity is attached to a single concept.
  • `{type_of_note}` : An indication of type of the note according to XKOS note types, e.g. “noteExclusions”, “noteGenerale”, “noteContenuCentral”, “noteContenuLimite”, etc.
  • `{note_version}` : the version of the note, e.g. “v1”, “v2”
  • `{language}` : the language code of the note, e.g. “fr”

  • http://id.insee.fr/codes/nafr2/classe/10.71/noteExclusions/v1/fr
  • http://id.insee.fr/codes/nafr2/classe/41.20/noteContenuCentral/v1/fr

Proposed URI pattern for Notes (timestamped)

{concept_URI}/{type_of_note}/{date}/{language}

Where:

  • `{concept_URI}` : the concept URI, as described above. The URI for the note is hierarchically below the URI of the concept having this note, since a note entity is attached to a single concept.
  • `{type_of_note}` : An indication of type of the timestamped note according to XKOS note types, typically a “changeNote” (“noteChangement”)
  • `{date}` : the date of the note, in an xsd:date format, e.g. “2021-09-22”
  • `{language}` : the language code of the note, e.g. “fr”

  • http://id.insee.fr/codes/cpfr2/sousCategorie/08.12.21/noteChangement/2010-02-01/fr
