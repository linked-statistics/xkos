The Best Practices

Description of a statistical classification

§5.1 of the XKOS specification states that

“In order to represent classifications and classification schemes, XKOS does not define specific classes but uses directly SKOS classes, adding to them different properties that specifically apply to classifications or classification schemes. Each major version of a classification is represented in XKOS by a `skos:ConceptScheme`.”
.

This best practices document gives further recommendations on how best to describe a `skos:ConceptScheme` that identifies a statistical classification, with mandatory, recommended and optional properties. Note that some constraints impose the presence of a property in English, hence the "lang" column with value "en" for certain properties

Mandatory, Recommended and Optional properties to describe statistical classifications
Property Lang. Card. Range Description
Mandatory properties
`skos:notation` 1..1 `xsd:string` The value must be the short name of the classification, with no language tag.
`skos:prefLabel` en 1..1 `rdf:langLiteral` The value must be the complete name of the classification scheme in english.
`dcterms:issued` 1..1 `xsd:date` The value must be the publication date of the classification scheme

Notes

§8.2 of the XKOS specification describes the different types of content notes introduced in XKOS and mentions the `xkos:plainText` property as a way to capture the raw content of the note. This best practice document gives additional recommendations on how best to describe the note entity, how to handle note versioning and timestamps, and clarifies the difference between `xkos:inclusionNote` and `xkos:caseLaw`.

Note structure

As indicated in the XKOS specification, and as a prerequisite for the versioning and timestamping of notes, instances of `xkos:ExplanatoryNote` should be used as value for the SKOS and XKOS note properties, and not literal values.

The instances of `xkos:ExplanatoryNote` should at least follow the structure given in the table below, in addition to other properties described below, depending if the note is timestamped or versioned.

General structure of an `xkos:ExplanatoryNote`
Property Card. Range Description
`xkos:plainText` 1..1 xsd:string The value of the note, in plain text (no HTML / XML markup). The literal is tagged with a proper language tag.
`dcterms:language` 1..1 xsd:language The language code of the note. This is a repetition of the language tag of the `xkos:plainText` attribute, that makes it easier to query/select notes in a certain language.
`rdf:value` 0..1 rdf:XMLLiteral If needed the value of the note containing HTML or XML markup.

Versioning notes and timestamping notes

As indicated in the introductory paragraph about classification major revisions and versions, the notes are the information that have fine-grained versions between each major revision of a classification.

We propose 2 different patterns of versioning for notes : versioning notes with a version number and a date range, and keeping the history of notes with a timestamp.

The first pattern to version notes implies that each new version of the note is given a version number and a date range. All notes, except `skos:changeNote` should follow this pattern : `skos:scopeNote`, `skos:definition`, `xkos:coreContentNote`, `xkos:additionalContentNote`, `xkos:exclusionNote`, `xkos:inclusionNote`, `xkos:caseLaw`. These notes should uses the properties listed in the table below.

Additional properties for an `xkos:ExplanatoryNote` when versioned
Property Card. Range Description
`dcat:version` 1..1 xsd:int The version number of the note. e.g. `"1"^^xsd:int`, `"2"^^xsd:int`, etc. Note that this version number will typically be used in the URI of the `xkos:ExplanatoryNote`.
`schema:startDate` 1..1 xsd:dateTime The date from which this note content is valid.
`schema:endDate` 0..1 rdf:XMLLiteral The date until which this note content was (or will be) valid, indicating it was (or will be) replaced with another note with a different text, and its version number increased by 1. This property is not set in the case if there is no known end date.

xkos:inclusionNote vs. xkos:caseLaw

When a framework classification is implemented it is generally accompanied by explanatory notes (represented using `skos:scopeNote` or its sub-properties). These explanatory notes can undergo some review procedures so that countries can provide feedback about their content. Once the new or updated/revised classification comes into force and for stability sake, the explanatory notes remain untouched until the classification is next revised or updated.

However between two revisions or updates questions arise from users about the classification of activities or products (goods or services) not explicitly mentioned in the official explanatory notes. Such questions are addressed to the custodians of the relevant classifications who then issue classification decisions after discussion with countries; as explanatory notes must remain untouched, these classification decisions are generally grouped under various names (e.g. "rulings", "classification decisions", "caselaws") and represented using the `xkos:caseLaw` property. When a framework classification is revised or updated theses caselaws are considered for inclusion in the official explanatory notes.

Below is an example of a caselaw relating to the NACE Rev. 2 classification. Rulings are displayed after `xkos:coreContentNote` and `xkos:additionalContentNote` properties but before `xkos:exclusionNote` property because such explanatory texts are labelled as inclusions and not as exclusions.

Levels

§6 of the XKOS specification specifies how to describe the levels in a classification, and how the property `xkos:organizedBy` can be used to link a level in a (specific revision of a) classification to the generic `skos:Concept` describing the generic kind of entries that this level regroups (e.g. the concept of an ISCO major group, of a NACE section, etc.).

We emphasize what XKOS already specifies :

  • the `skos:ConceptScheme` uses the property `xkos:levels` to point to an `rdf:List` that contains an ordered list of instances of `xkos:ClassificationLevels` in this classification. The list is ordered from the highest level to the lowest. These classification levels represent the grouping of the concepts in this particular revision of the classification.
  • Each instance of `xkos:ClassificationLevel` uses `xkos:organizedBy` to refer to a `skos:Concept` describing the “abstract” kind of entries at this level of the classification, independently of the revision of the classification.

We give additional best practices :

  1. the `skos:Concept`s describing the “abstract” kind of entries should be organized hierarchically using `skos:broader` / `skos:narrower` relationships.
  2. they should not be blank nodes.
  3. these `skos:Concept` should belong to a `skos:ConceptScheme` that is the list of levels in this classification, in general (not a particular revision of the classification).
  4. these `skos:Concept` can be declared in the same file as the `skos:ConceptScheme` of the classification, and thus will be repeated with each major revision of the classification.
# The classification revision with an ordered list of its classification levels
ex:NACE-R2 a skos:ConceptScheme;
  skos:prefLabel “NACE-r2”@en ;
  xkos:levels (
    ex:NACE-R2_Section,
    ex:NACE-R2_Division,
    ex:NACE-R2_Group,
    ex:NACE-R2_Class
  )
.

# Description of classification levels in this particular revision
# Each having an xkos:organizedBy that points to levels of the NACE
# Independantly of the revision
 
ex:NACE-R2_Section a xkos:ClassificationLevel ;
  xkos:organizedBy ex:NACE_Section .
 
ex:NACE-R2_Division a xkos:ClassificationLevel ;
  xkos:organizedBy ex:NACE_Division .
 
ex:NACE-R2_Group a xkos:ClassificationLevel ;
  xkos:organizedBy ex:NACE_Group .
 
ex:NACE-R2_Class a xkos:ClassificationLevel ;
  xkos:organizedBy ex:NACE_Class .
 
 
# Description of abstract levels of the NACE
# Organized hierarchically, and all belonging to a ConceptScheme
 
ex:NACE_Section a skos:Concept ;
  skos:inScheme ex:NACE_ItemTypes ;
  skos:topConceptOf ex:NACE_ItemTypes ;
  skos:prefLabel “NACE Section”@en ;
.
 
ex:NACE_Division a skos:Concept ;
  skos:inScheme ex:NACE_ItemTypes ;
  skos:prefLabel “NACE Division”@en ;
  skos:broader ex:NACE_Section ;
.
 
ex:NACE_Group a skos:Concept ;
  skos:inScheme ex:NACE_ItemTypes ;
  skos:prefLabel “NACE Group”@en ;
  skos:broader ex:NACE_Division ;
.
 
ex:NACE_Class a skos:Concept ;
  skos:inScheme ex:NACE_ItemTypes ;
  skos:prefLabel “NACE Class”@en ;
  skos:broader ex:NACE_Group ;
.
 
ex:NACE_ItemTypes a skos:ConceptScheme ;
  skos:hasTopConcept ex:NACE_Section ;
  skos:prefLabel “NACE Item Types”@en ;
.

Types of the classification entries

The XKOS specification does not define a specific class to represent classification items but rather directly relies on `skos:Concept`. This best practices document gives further recommendations on how the classification items can be typed according to their levels.

Type the classification items of a given classification

A best practice is to declare a new subclass of `skos:Concept` as the type of the classification items for a given revision of a classification. For example the class “ex:NACE-R2_Item” could be declared as a subclass of `skos:Concept` to represent the set of all items in the 2nd revision of the NACE classification. Introducing this class allows to constrain the range of certain properties on this class.

This new subclass of skos:Concept shall be linked to the skos:ConceptScheme with either of these 2 modelling patterns :

  1. an `rdfs:seeAlso` property, in both direction (from the `skos:ConceptScheme` to the new sub-class of `skos:Concept`, and from the new subclass of `skos:Concept` to the `skos:ConceptScheme`)
  2. a formal OWL equivalence between the subclass of `skos:Concept` and a restriction on the property `skos:inScheme`, with an `owl:hasValue` equal to the `skos:ConceptScheme` URI. For example here is how one would declare the equivalence between a new subclass of skos:Concept "NACE-R2_Item" and the set of `skos:Concept` that have a `skos:inScheme` equal to the corresponding scheme URI : 
    Class: ex:NACE-R2_Item

    EquivalentTo:
        skos:inScheme value ex:nace-r2

While the first option is more direct and readable, the second one is more formal.

This new class is expected to be declared in the same file as the `skos:ConceptScheme` representing the classification.

Both these options are illustrated in the example below.

# The ConceptScheme
ex:NACE-R2 a skos:ConceptScheme;
  skos:prefLabel “NACE-r2”@en ;
  # See also the Class
  rdfs:seeAlso ex:NACE_Item ;
.
 
# The Class, subClassOf skos:Concept
ex:NACE-R2_Item a owl:Class;
  rdfs:subClassOf skos:Concept ;
  skos:prefLabel “NACE R2 Item”@en ;
  # See also the ConceptScheme
  rdfs:seeAlso ex:NACE-R2 ;
  # Formal OWL equivalence to the set of Concepts belonging to the ConceptScheme
  owl:equivalentClass [
    a owl:Restriction ;
    owl:onProperty skos:inScheme ;
    owl:hasValue ex:NACE-R2 ;
  ] ;
.

Note that items inside the classification are not expected to be explicitely typed with this new class; they should only be typed with `skos:Concept`. The declaration of this class is only useful so that it can be assigned as the range of some properties.

Type classification items according to their level

§6 of the XKOS specification describing classification levels states the following :

"In XKOS, a classification level is defined as a specialization of `skos:Collection` named `xkos:ClassificationLevel`. (...) Individual `skos:Concept` objects are related to the `xkos:ClassificationLevel` to which they belong by the usual `skos:member` property (going from the level to the classification item)."

This best practice document suggests to go one step further, and to actually consider the classification levels also as OWL classes (hence with 2 types : `xkos:ClassificationLevel` and `owl:Class`), declared as subclass of the dedicated class that represent the set of all items of the classification, as described in previous section.

Following the example given above, the following levels of the NACE-R2, in addition to being `xkos:ClassificationLevel`, would be typed as `owl:Class` and declared subclasses of `ex:NACE-R2_Item` :

  • ex:NACE-R2_Section
  • ex:NACE-R2_Division
  • ex:NACE-R2_Group
  • ex:NACE-R2_Class

These classes would be declared as direct subclasses of `ex:NACE-R2_Item`, as the set of e.g. NACE sections is a subset of the set of all NACE items. However these classes would not be declared as subclasses of each other, as it is not true to state that e.g. the set of all NACE R2 divisions is a subset of all NACE R2 sections.

To capture the intent that instances of NACE R2 divisions are expected to be narrower concepts of instances NACE R2 sections, these entries are referenced in an ordered list by the `xkos:levels` property on the `skos:ConceptScheme`.

This is illustrated in the following example.

# The classification revision with an ordered list of its classification levels
ex:NACE-R2 a skos:ConceptScheme;
  skos:prefLabel “NACE-r2”@en ;
  xkos:levels (
    ex:NACE-R2_Section,
    ex:NACE-R2_Division,
    ex:NACE-R2_Group,
    ex:NACE-R2_Class
  )
.

# Description of classification levels in this particular revision
# Each having an xkos:organizedBy that points to levels of the NACE
# Independantly of the revision
 
ex:NACE-R2_Section a xkos:ClassificationLevel, owl:Class ;
  rdfs:subClassOf ex:NACE-R2_Item ;
  xkos:organizedBy ex:NACE_Section .
 
ex:NACE-R2_Division a xkos:ClassificationLevel, owl:Class ;
  rdfs:subClassOf ex:NACE-R2_Item ;
  xkos:organizedBy ex:NACE_Division .
 
ex:NACE-R2_Group a xkos:ClassificationLevel, owl:Class ;
  rdfs:subClassOf ex:NACE-R2_Item ;
  xkos:organizedBy ex:NACE_Group .
 
ex:NACE-R2_Class a xkos:ClassificationLevel, owl:Class ;
  rdfs:subClassOf ex:NACE-R2_Item ;
  xkos:organizedBy ex:NACE_Class .

# The common parent class that represents the sets of all entries of the classification
# linked to the ConceptScheme with an rdfs:seeAlso and a formal OWL equivalence

ex:NACE-R2_Item a owl:Class;
  rdfs:subClassOf skos:Concept ;
  skos:prefLabel “NACE R2 Item”@en ;
  rdfs:seeAlso ex:NACE-R2 ;
  owl:equivalentClass [
    a owl:Restriction ;
    owl:onProperty skos:inScheme ;
    owl:hasValue ex:NACE-R2 ;
  ] ;
.
 
# Description of abstract levels of the NACE
# Organized hierarchically, and all belonging to a ConceptScheme
 
ex:NACE_Section a skos:Concept ;
  skos:inScheme ex:NACE_ItemTypes ;
  skos:topConceptOf ex:NACE_ItemTypes ;
  skos:prefLabel “NACE Section”@en ;
.
 
ex:NACE_Division a skos:Concept ;
  skos:inScheme ex:NACE_ItemTypes ;
  skos:prefLabel “NACE Division”@en ;
  skos:broader ex:NACE_Section ;
.
 
ex:NACE_Group a skos:Concept ;
  skos:inScheme ex:NACE_ItemTypes ;
  skos:prefLabel “NACE Group”@en ;
  skos:broader ex:NACE_Division ;
.
 
ex:NACE_Class a skos:Concept ;
  skos:inScheme ex:NACE_ItemTypes ;

Describing the relation between an international and a national classification

The relationship between a classification and its derivation is partly described by the correspondence table which, for a given level, maps the items of the two nomenclatures.

Specific information can also be expressed on the levels to characterise them in relation to each other. Thus, for some derivations an additional level is defined or the granularity of the level is finer, for example to provide more precision at national level. It is suggected that the relationship between the finer level of the derived classification and the more aggregated level of the reference classification be expressed with a `skos:broadMatch` predicate.

Below is an example where the level is thinner in the NACE compare to ISIC (NACE being an EU-level extension of the world-level ISIC classification).

ex:NACE-2-Group a xkos:ClassificationLevel ;
  xkos:organizedBy ex:NACE_Group .
 
ex:NACE_Group a skos:Concept ;
skos:prefLabel "NACE Group"@en ;
skos:broadMatch ex:ISIC_Group .
 
ex:ISIC_Group a skos:Concept ;
skos:prefLabel "ISIC Group"@en .

XKOS does not propose a property to indicate that a (national) classification refines another (international) one. However, it would be possible to state a `prov:wasDerivedFrom` property on the concept scheme of the derived classification.

Evolution of classifications over time

Major versions of classifications

When a complete revision of a classification happens, a new major version of a classification is produced. The recommendation is that all the URIs of the concepts and levels are recreated and the relation between the classification items is given by a correspondence table as presented in the corresponding section of this document. The former classification should have the property `owl:deprecated` set to “true” and its period of validity updated in a `dcat:temporal` property.

Linking major versions of classifications

In the situation of managing classifications over time, links between successive major versions of classifications can be created at the `skos:ConceptScheme` level by using `xkos:supersedes` that conveys the idea that the value of this property is a deprecated classification.

Use of PROV-O

§5.1 of the XKOS Specification states

“Readers should also be aware that the Provenance Ontology, which was not available at the time of writing of this specification, introduces the related notion of "entity specialization" (`prov:specializationOf`). This can be an alternative approach for describing the versioning of classifications.”

The following table gives more details on how XKOS properties are related to [[PROV-O]]. The use of [[PROV-O]] properties in combination with XKOS properties is encouraged to promote wider interoperability of links between classifications.

Relations between XKOS and PROV-O
XKOS Property or Path Alignement PROV-O property
`xkos:supersedes` `rdfs:subPropertyOf` `prov:wasRevisionOf`
`xkos:follows` `rdfs:subPropertyOf` `prov:wasDerivedFrom`
`^xkos:variant` `rdfs:subPropertyOf` `prov:wasDerivedFrom`
`^xkos:targetConcept/xkos:sourceConcept` `rdfs:subPropertyOf` `prov:wasDerivedFrom`
`^xkos:belongsTo` `rdfs:subPropertyOf` `prov:specializationOf`

Note that in the case of a `xkos:variant`, the corresponding [[PROV-O]] property is `prov:wasDerivedFrom` instead of `prov:wasRevisionOf`. The existence of a variant does not imply the original version is deprecated. There will be an overlap between the validity time period of the variant version and the reference one.

Publishing statistical classification data

While the XKOS specification introduces useful properties to describe statistical classifications, it does not give recommendations on how to publish statistical classification data. This section explains how to use [[DCAT]] to describe and publish the datasets of statistical classifications and correspondences.

Datasets of classifications

It is recommended to use DCAT V2 to describe and disseminate metadata associated with classifications.

Each revision of a classification should correspond to one `dcat:Dataset`. For instance, in the example below, the classification NAF Rev1 and NAF Rev2 are separated in distinct `dcat:Dataset`s with different version information. Each classification’s dataset belongs to a `dcat:Catalog`.

Each dataset should have downloadable files in the necessary formats for both human and machine agents. In particular, statistical classifications should be provided for download in an RDF file based on [[SKOS]] and [[XKOS]] vocabularies, containing the full content of the classification. In the example, CSV and RDF (SKOS) formats are provided. The `dct:conformsTo` property should be set to the URI of the [[XKOS]] vocabulary : http://rdf-vocabulary.ddialliance.org/xkos.

For practical reasons it is important to have direct access to the latest version of a classification for directly retrieving the updated notes. However, for reasons of nomenclature expertise it is also important to have access to changes in notes that have affected the nomenclature. It is therefore recommended to have access to the history of modifications of all notes within the revision of the classification included in a `dcat:Dataset`. These versions containing the history of all notes are considered as different distributions of the datasets. They should be described by specific properties. Here we use `rdfs:comment` for describing and discriminating the distributions, given a lack of other alternatives. Datasets of correspondences

Each correspondence table shall also be provided as a distinct Dataset in the catalog. This dataset can be linked with the dataset of the source classification and the dataset of the target classification with :

  • A generic `dct:relation` property, that is expressed both from the source and target dataset to the correspondence table dataset, and from the correspondence dataset to its source and target.
  • A `dcat:Relationship` that indicates more precisely the role of each dataset, either source or target in the correspondence table. The role of each dataset is captured by the `dcat:hasRole` property, with one of these 2 values :
    • `xkos:correspondenceFromRole`
    • `xkos:correspondenceToRole`
    The exact URIS for roles have been proposed to XKOS but are not part of the standard at the time of writing of this document, and so are subject to change.

The example below presents the correspondence between Naf rev2 and Naf rev1 classification.

# The correspondence as a dcat:Dataset, with links to the source and target datasets
# from the previous example
ex:CorrespondenceTableNafrev1Nafrev2Dataset a dcat:Dataset ;
  dcat:distribution ex:CorrespondenceTableCsv ;
  dcat:distribution ex:CorrespondenceTableRdf ;
  # Qualified relation to the source dataset
  dcat:qualifiedRelation [
    a dcat:Relationship ;
    dct:relation ex:NafRev1Dataset ;
    dcat:hadRole xkos:correspondenceFromRole ;
  ] ;
  # Qualified relation to the target dataset
  dcat:qualifiedRelation [
    a dcat:Relationship ;
    dct:relation ex:NafRev2Dataset ;
    dcat:hadRole xkos:correspondenceToRole ;
  ] ;

URI patterns Best Practices

Proposed URI pattern for classifications

{scheme}://{authority}/{all_classifications}/{classification_naming_context}/{concept_scheme_id}

Where:

  • `{scheme}`: is the URI scheme, typically “http” or “https”.
  • `{authority}` : the domain name of the authority attributing the URI to the resource. This guarantees the unicity of the URI at Web scale. Note that the domain name can be a dedicated subdomain of the more general domain name of the authority e.g. “data.acme.org” or “id.acme.org”.
  • `{all_classifications}` : is a fixed component in the URI indicating that what follows pertains to the identification of an entity in the scope of a classification, as opposed to an entity in another domain, like statistical data, document metadata, etc. Typical examples are “codes”, “classifications”, etc.
  • `{classification_naming_context}` : an identifier for the naming context that will contain the different components of this classification; This component typically is based on the classification id in one of its specific revisions. Examples are “nafr2”, “pcs1982”, etc.
  • `{concept_scheme_id}` : an identifier for the concept scheme that represents a revision of the classification.

Given that the french NAF rev2 is identified with the classification_naming_context `nafr2` and the concept_scheme_id `naf`, the corresponding URI would be `http://id.insee.fr/codes/nafr2/naf`

Proposed URI pattern for classification levels

{scheme}://{authority}/{all_classifications}/{classification_naming_context}/{level}

Where:

  • `{scheme}`, `{authority}`, `{all_classifications}` and `{classification_naming_context}` components have the same meaning as the in the URI pattern for the classification;
  • `{level}` : an identifier for a level in this classification. Some implementations have chosen the plural forms, e.g. “sections”, “classes”, “divisions”, etc.

Given that the french NAF rev2 contains levels that are “sections”, “divisions”, “groupes”, “classes”, “sousClasses”, the corresponding URI would be :

  • http://id.insee.fr/codes/nafr2/sections
  • http://id.insee.fr/codes/nafr2/divisions
  • http://id.insee.fr/codes/nafr2/groupes
  • http://id.insee.fr/codes/nafr2/classes
  • http://id.insee.fr/codes/nafr2/sousClasses

Proposed URI pattern for classification entries (Concepts)

{scheme}://{authority}/{all_classifications}/{classification_naming_context}/{level}/{concept_id}

Where:

  • `{scheme}`, `{authority}`, `{all_classifications}` and `{classification_naming_context}` components have the same meaning as in the URI pattern for the classification;
  • `{level}` : an identifier for the level of the concept. It is key to understand that the level is an intrinsic part of the concept identity in statistical classifications. A concept cannot be moved in a given revision of a classification. Hence the level is part of the concept URI. Implementations have chosen to use the singular form of the level, thus a different stem from the URI of the corresponding level. Examples are “section”, “class”, “division”, etc.
  • `{concept_id}` : an identifier for the specific concept within the classification; the format of the identifier varies depending on the classification and the level. Examples are “A”, “1963”, “10.73”, etc.

Given that the french NAF rev2 contains levels identified with the level codes “section”, “division”, “groupe”, “classe”, “sousClasses”, corresponding examples of concepts at each of these levels are :

  • http://id.insee.fr/codes/nafr2/section/A
  • http://id.insee.fr/codes/nafr2/division/01
  • http://id.insee.fr/codes/nafr2/groupe/01.1
  • http://id.insee.fr/codes/nafr2/classe/01.16
  • http://id.insee.fr/codes/nafr2/sousClasse/01.16Z

Proposed URI pattern for Labels

{concept_URI}/{label_id_length_language}

Where:

  • `{concept_URI}` : the concept URI, as described above. The URI for the label is hierarchically below the URI of the concept having this label, since a label entity is attached to a single concept.
  • `{label_id_length_language}` : a component identifying the label, typically composed with a fixed part, the label length and the languages. Examples are “label40fr” or “label65en”

  • http://id.insee.fr/codes/nafr2/classe/01.16/label40fr
  • http://id.insee.fr/codes/nafr2/classe/01.16/label65fr

Proposed URI pattern for Notes (versioned)

{concept_URI}/{type_of_note}/{note_version}/{language}

Where:

  • `{concept_URI}` : the concept URI, as described above. The URI for the note is hierarchically below the URI of the concept having this note, since a note entity is attached to a single concept.
  • `{type_of_note}` : An indication of type of the note according to XKOS note types, e.g. “noteExclusions”, “noteGenerale”, “noteContenuCentral”, “noteContenuLimite”, etc.
  • `{note_version}` : the version of the note, e.g. “v1”, “v2”
  • `{language}` : the language code of the note, e.g. “fr”

  • http://id.insee.fr/codes/nafr2/classe/10.71/noteExclusions/v1/fr
  • http://id.insee.fr/codes/nafr2/classe/41.20/noteContenuCentral/v1/fr

Proposed URI pattern for Notes (timestamped)

{concept_URI}/{type_of_note}/{date}/{language}

Where:

  • `{concept_URI}` : the concept URI, as described above. The URI for the note is hierarchically below the URI of the concept having this note, since a note entity is attached to a single concept.
  • `{type_of_note}` : An indication of type of the timestamped note according to XKOS note types, typically a “changeNote” (“noteChangement”)
  • `{date}` : the date of the note, in an xsd:date format, e.g. “2021-09-22”
  • `{language}` : the language code of the note, e.g. “fr”

  • http://id.insee.fr/codes/cpfr2/sousCategorie/08.12.21/noteChangement/2010-02-01/fr
